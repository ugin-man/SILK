# Taxonomy and views

## Purpose

Collections help a human find canonical subjects. They are not containers for pieces of a subject and are not a universal theory of what can exist.

## Primary collection decision

Choose the physical collection by asking:

- Where would a reader reasonably look for this subject?
- What is the subject primarily being treated as in this world?
- Which location keeps neighboring files understandable without granting shared traits?

Use secondary collections for other useful paths. Do not duplicate the canonical file.

## Do not classify facets as subjects

Origin, roots, powers, appearance, worship, political use, history, failure, and current condition are usually facets inside a subject's canonical file. Do not create `deity_origins`, `deity_powers`, and `deity_history` files for one deity unless an origin event, power mechanism, or historical transformation has independent identity and multi-subject consequences.

## Unknown concepts

When a new concept does not fit existing collections:

1. decide whether it is an independent subject;
2. test whether an existing collection can hold it without semantic distortion;
3. add a provisional collection only when it improves navigation;
4. give the collection a role and reason, not a list of inherited fields;
5. revisit after more subjects exist;
6. merge, rename, split, or retire the collection without changing subject IDs.

Separate machine ID from display label. Renaming a human-facing collection label does not require changing stable subject IDs.

One singular load-bearing subject may justify a collection. A large number of files does not automatically justify one.

## Collection audit

Flag:

- one subject fragmented across facet collections;
- a collection that acts as a dumping ground;
- two collections that differ only in name;
- collections based on file format or workflow stage rather than world meaning;
- an important subject hidden under an unintuitive primary collection;
- empty or speculative collections;
- a collection whose members are assumed to share traits;
- category trees so deep that users cannot find a canonical file.

## Views

Generate:

- collection-first catalog with approved, pending, and discarded sections;
- status-first view with collection and maturity;
- aliases and redirects from old paths or names;
- relation map with meaningful summaries;
- optional chronology, geography, belief, or system views only when useful.

Views may repeat titles and summaries. They must not duplicate canonical lore.

## Reader test

A reader should be able to:

1. find a subject through a plausible collection or alias;
2. see whether the user approved it;
3. open one canonical file and understand the subject as a whole;
4. follow summarized related items;
5. return to a generated view without encountering a second source of truth.
