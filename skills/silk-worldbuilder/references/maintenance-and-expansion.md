# Maintenance and expansion

## Reopen deliberately

When the user asks to expand an internally complete or user-approved world:

1. increment package revision;
2. preserve user-approved and locked canon;
3. record the requested expansion and affected scope envelope;
4. set world status to `building` or `changes_requested`;
5. open hypotheses and work packets under pending status;
6. identify maturity gates and audits invalidated by the change.

Do not regenerate the world from Level 0. Reopen the smallest affected subjects and world level, then run outward impact review.

## Expansion types

- deepen an existing canonical subject;
- add a new independent subject;
- extend spatial, temporal, social, or ontological scope;
- revise user-approved canon with explicit permission;
- add a relation or junction;
- reinterpret a disputed claim;
- replace generated provisional material;
- import V3 story candidates as hypotheses.

## Scope expansion

Expanding the envelope creates coverage obligations. A new continent, era, plane, or species cannot remain a name list. Derive new roots before weaving them into old ones, then audit convergence and anthology soup.

## Approved canon impact

If expansion contradicts approved canon, preserve the approved version and present options. If the user authorizes revision, record old claim, new claim, reason, affected IDs, approval scope, and migration action.

## Revalidation

Run subject and affected-system review during work. Before returning to internally complete, rerun both full audit routes. Previous clean streak does not survive a load-bearing expansion.

## Archive and revision

Keep the current package readable. Decision history records prior states; canonical files show current canon. Do not retain duplicate world trees for every small revision unless external version control handles snapshots.
